Description of design efforts: In week 5, I set up the microcontroller, the STM32F407. Use cube and HAL library to generate the skeleton of the firmware that we are going to use. I thought about using STM32L4 and STM32F0 family, but F4 has more peripherals compared to F0 and less occupied GPIO for those peripherals compared to L4. Although I have experience in programming F0, and the library are quite similar, it still take me some time to get familiar with the library code and peripheral setting. That reminds me again that if knowledge is not being used all the time, it's quite easy to lose them. I start with a TIM interrupt and blink the LED, then go to PWM to take the work from Arduino, since previously the ESC module is controlled by the PWM signal from Arduino. At first, I found that the voltage of PWM signal is 3V which is not high enough for ESC, so I decided to use a N-MOSFET to create a 5V PWM signal with the same duty cycle. However, it either decrease the voltage or invert the duty cycle which confused me a lot. Later on, I check what I learned in ECE255 and get to know that a MOSFET can only create an inverse signal because of its intrinsic mechanism, and in the circuit, a load is very necessary to drain that voltage. As an engineering student, I feel ashamed that I forgot all of those knowledge that I should command, but in the meanwhile, I feel fortunate that I can still trace them back and gain them again.

Another major task is to set up the WiFi module, which is the most important communication bridge in our project. The ESP8266 talk to machine via UART. For convenience, I used the UART module connected to PC to set up connect WiFi first. During that process, I learned that the private IP address that the router gives to a device could change in few days, and I have to assign the ESP8266 a specific IP address to prevent it from changing. After that has been done, open the ESP8266 to the public internet is also a significant step. After the private IP address device is forwarded to public, every with internet access can sent signal to the device through public IP address. All of these are working fine with UART module and Putty interface, but when switching to the STM32F4 board, there is some issue with receiving the right response, and I will keep working on that later on.

Next week I'm going to fix the WiFi module problem first, and then make GPS work with the STM32F4.












































































Week 6:

Date: 9-28-2018
Total hours: 18
Description of design efforts:

In week6, I keep working on the WiFi module and the communication with the user interface. At the start, I used HTTP protocol and created a server which can be accessed with public internet. It is a very easy access, one can even use IE explorer to get a precise feedback, but the communication rate is slow because it delay for other information that is not needed in our case. Then I looked for a more convenient and economic way to communicate with the user interface. TCP protocol comes to be the best solution. With TCP protocol, the transmission rate is much faster, and neat messages are sent. The ESP8266 module can either create a server or be a client to access an external server using TCP protocol. Since the public address could change when creating a new server on the user interface with different internet connection, but the WiFi source we have on the boat is fixed. Thus making ESP8266 as a server is a clever choice. So far, the TCP signal from user interface can adjust the PWM signal to the ESC and motor rotation speed.

Another major step I have is to record the voltage through ADC peripheral. It seems to be an easy step since we’ve all known how to control the brightness of an LED by a potentiometer through ATD and DTA. However, in our project we have a lot voltage need to be monitored through ADC. There are five break beam sensor made by LDR circuitry, whose voltage need to be monitored to see the fullness of the storage room. Besides we also have three battery groups and probably 9 cells in total to drive the circuity and motors, and the voltage of each of the cell needs to be monitored. We need over 10 ADC channels in total. With that amount of channels open, the sampling rate, interrupt rate and memory used are all need to be reconsidered. Now, the micro can transfer multiple channel inputs without interference, after I set the reasonable sampling cycles and open DMA mode.

My next step is to get read from GPS and Compass. So far, the GPS UART transfer is done neatly, but the NMEA form is hard to parse into float coordinates. I will work on that next week.












































Week 7:

Date: 10-5-2018
Total hours: 13
Description of design efforts:

In week7, I work with Peiyuan together on the software part, and have done GPS communication properly. On the microcontroller, there are two UART peripheral open for GPS and Wi-Fi module. Initially, the buffer of both UART is single byte, which means, they read character by character. However, the GPS message transmission is very fast and leads to message leaking on the Wi-Fi communication. Then I change the buffer from a single byte to 512 byte, because our system does not require an extreme high speed refresh rate of GPS coordinates. A large buffer will discard some overlapped message but can at least extract one intact useful message with all the information we need. With this method, both GPS and Wi-Fi module can talk to micro well.

Another aspect of the communication is between the micro and the user interface. On the micro side, every time it reboots, ESP8266 will be reset and get ready to be connected, once “OK” message is received. It is constantly hearing from the user interface, and once receive “command” from the user interface, it sends a series of information including GPS, compass, battery, and fullness of storage to the user interface. If the interface cannot connect to ESP8266, or any error or exception happens during the communication, it will print “connection error”. If interface successfully sends the message but cannot hear from the microcontroller, it will try to reconnect the Wi-Fi module and keep sending a message.

Besides software part, I also help design the MOSFET switching circuit, and test the IC and other components we have. Next week I will prepare for the design review and test the Compass module.

Week 12:

Date: 11-9-2018
Total hours: 19
Description of design efforts:
In week 12, I did two major things. One is calibrating the compass, another is making the boat with fiberglass in BIDC.

On the compass part, I soldered the break out board of the compass, and tested the function of compass module successfully. I also took the whole compass module including the micro outside of the building to test. Todd told us that compass requires a calibration step before putting into use. Until I took the module outside and found the difference of performance in different environment, I didn’t fully understand that why calibration is necessary. Afterwards, I found that if compass turns 360 degree around, the module returns bunches of points on Cartesian coordinate system which show as a circle. However, the central of the circle is not exactly <0,0>, and the center of the circle will vary in different environment. Thus, to calibrate the center of circle is necessary to know the angle.

Besides that firmware part, I also spent two days in BIDC to make the boat. The fiberglass combined with epoxy resin provides a super robust and hard shell covered on the hull of the boat. The most important part in the manufacturing process is safety, especially when cutting the stiffened fiberglass, because the dust of fiberglass is irritant, and if inhaled by human beings, would cause a lot of damage to the respiratory system. We were very careful when we did hand on cutting and sanding the fiberglass.

Next week, I will keep making the boat and solder the PCB, and hopefully assemble the whole boat.
